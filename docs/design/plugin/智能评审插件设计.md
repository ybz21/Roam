# 智能评审插件设计：Codex + Claude 互审机制

> 状态: **设计草案**　日期: 2026-07-03
>
> 定位: 在 Roam / ttmux 上构建一个智能化插件，让 Codex、Claude 以及后续其他 Agent 围绕同一任务、同一 diff、同一 swarm 成员产出进行互相 review、争议归因、修复闭环和质量门禁。

---

## 1. 目标

通用插件机制的第一落点不应该是“让第三方加菜单”，而应该是更贴近 Roam 的核心场景：

> **多个 AI coding agent 在同一开发机上协作，并且互相审查、互相纠偏。**

智能评审插件要解决的问题：

- Claude 写了一批代码，Codex 自动 review，指出风险、缺测试、破坏兼容性的地方。
- Codex 写了一批代码，Claude 从产品意图、可维护性、实现完整度上反审。
- 两个 Agent 对同一个问题给出不同判断时，由插件组织“证据对齐”，要求双方引用文件、命令输出、测试结果，而不是空泛争论。
- review 结果不只是聊天文本，而是转成 swarm 看板任务、阻塞项、修复建议、质量门禁状态。
- 人类可以在 Web 控制台看到“谁写的、谁审的、争议是什么、是否已修复、是否可合并”。

---

## 2. 产品形态

插件名称建议：

```text
review-mesh
```

中文可叫：

```text
智能互审 / Review Mesh / Agent 互审网
```

核心体验：

```bash
ttmux plugin run review-mesh.review --target HEAD
ttmux plugin run review-mesh.review --swarm <swarm>
ttmux plugin run review-mesh.review-member <swarm> <member>
ttmux plugin run review-mesh.resolve --finding <id>
```

Web 上的入口：

- swarm 详情页：`发起互审`
- 成员详情页：`让 Codex review Claude` / `让 Claude review Codex`
- diff 面板：显示 Findings、争议、修复状态
- 看板：自动生成 `review-blocker`、`needs-fix`、`verified` 卡片

Agent 侧的入口：

- Leader 可以说：“对 frontend 成员做 Codex review。”
- Reviewer Agent 可以调用插件工具读取 diff、运行测试、发布 findings。
- Fixer Agent 可以领取具体 finding 并提交修复，再触发复审。

## 3. 角色模型

智能互审不是简单开两个终端互相说话，而是有角色分工。

| 角色 | 推荐 Agent | 职责 |
|---|---|---|
| Author | Claude 或 Codex | 实现功能、解释设计意图、响应 review |
| Reviewer | Codex 或 Claude | 从代码风险、测试、边界条件、回归上审查 |
| Product Reviewer | Claude | 检查是否满足用户目标、交互和文案是否合理 |
| Risk Reviewer | Codex | 检查工程风险、并发、安全、错误处理、测试缺口 |
| Judge | 可配置 | 当双方冲突时做证据裁决，要求引用来源 |
| Fixer | 原 Author 优先 | 按 finding 修复，不允许无关重构 |
| Verifier | 任意 Agent 或 task | 跑质量门禁、测试、截图、复现命令 |

默认推荐矩阵：

| 产出来源 | 第一 reviewer | 第二 reviewer | 理由 |
|---|---|---|---|
| Claude 写代码 | Codex | 可选 Claude 自审 | Codex 更适合作为严格代码审查者 |
| Codex 写代码 | Claude | 可选 Codex 自审 | Claude 更适合从需求表达和产品完整性反审 |
| 前端 UI | Claude + 截图验证 | Codex | Claude 看体验，Codex 看实现和测试 |
| 后端/CLI | Codex | Claude | Codex 看工程风险，Claude 看使用路径 |
| 文档/产品设计 | Claude | Codex | Claude 看叙事，Codex 看可实施性和一致性 |

## 4. 工作流

### 4.1 单次互审

```text
1. 捕获变更范围
   - git diff / git show / swarm member output
   - 相关文件列表
   - 用户目标和验收标准

2. 生成 Review Brief
   - 背景
   - 改动摘要
   - 明确审查维度
   - 禁止事项：不要重写、不要跑偏、必须引用证据

3. 启动 Reviewer Agent
   - Codex review Claude 产出
   - 或 Claude review Codex 产出

4. 结构化收集 Findings
   - severity
   - file / line
   - claim
   - evidence
   - suggested fix
   - test gap

5. Author 响应
   - accept：承认并修复
   - reject：给出证据反驳
   - clarify：需要人类决策

6. Fixer 修复
   - 对 accepted finding 做最小改动

7. Verifier 复验
   - 跑测试/质量门禁
   - Reviewer 复审 finding 是否关闭
```

### 4.2 Swarm 内自动互审

swarm 生命周期中插入 hook：

```text
member.completed
  -> review-mesh.createReviewJob(member)
  -> spawn reviewer agent
  -> reviewer findings 写入 board
  -> blocker 阻止 swarm.integrating

finding.fixed
  -> review-mesh.recheckFinding(finding)
  -> verified 后关闭 blocker

swarm.integrating
  -> review-mesh.finalCrossReview()
  -> 生成最终质量报告
```

### 4.3 双 Agent 辩论不是默认路径

默认不让 Agent 长时间“辩论”。互审插件默认只允许三轮：

```text
Reviewer 提出 finding
Author 响应 accept/reject/clarify
Judge 裁决 accept/reject/human-needed
```

超过上限自动升级给人类，避免 token 消耗和无效争论。轮次上限可配置：设置页
`rounds` 字段（留空=3，硬上限 20），或单次陪跑用 `review-mesh.watch --rounds N`
覆盖。

## 5. Finding 数据结构

```json
{
  "id": "finding-20260703-001",
  "swarm": "plugin-system",
  "member": "frontend",
  "authorAgent": "claude",
  "reviewerAgent": "codex",
  "status": "open",
  "severity": "high",
  "category": "bug",
  "title": "Plugin iframe can call unrestricted backend endpoint",
  "claim": "The new plugin iframe bridge does not validate plugin id before forwarding invoke requests.",
  "evidence": [
    {
      "file": "backend/plugin/bridge.go",
      "line": 87,
      "quote": "proxyInvoke(pluginID, payload)"
    }
  ],
  "impact": "A malicious plugin UI may invoke another plugin's command if it can guess the id.",
  "suggestedFix": "Bind iframe origin/session to plugin id and verify it on every invoke.",
  "testGap": "Add a bridge authorization test for mismatched plugin id.",
  "authorResponse": null,
  "judgeDecision": null,
  "createdAt": "2026-07-03T12:00:00+08:00",
  "updatedAt": "2026-07-03T12:00:00+08:00"
}
```

状态机：

```text
open -> accepted -> fixed -> verified -> closed
open -> rejected -> judged-rejected -> closed
open -> needs-human
open -> duplicate
open -> wontfix
```

严重级别：

| severity | 含义 | 默认行为 |
|---|---|---|
| critical | 安全、数据损坏、无法启动、破坏主流程 | 阻塞合并/集成 |
| high | 明确 bug、回归、重要测试缺失 | 阻塞集成 |
| medium | 边界问题、可维护性风险 | 进入修复队列 |
| low | 命名、文档、轻微一致性 | 不阻塞 |
| note | 观察项 | 只记录 |

### 6.2 CLI

### 6.3 存储

```text
~/.local/share/ttmux/reviews/
├── review-jobs/
│   └── <job-id>.json
├── findings/
│   └── <finding-id>.json
├── reports/
│   └── <swarm>-<timestamp>.md
└── transcripts/
    └── <job-id>/
        ├── brief.md
        ├── reviewer-output.md
        ├── author-response.md
        └── judge-output.md
```

原则：

- Finding 用 JSON，便于 Web、CLI、Agent 读取。
- Agent 原始输出保留为 transcript，便于审计和追溯。
- 报告用 Markdown，便于贴到 PR、Issue 或任务总结。

## 7. Agent 启动策略

### 7.1 不直接假设某个 Agent 永远可用

插件应维护 Agent provider 注册表：

```json
{
  "providers": [
    {
      "id": "codex",
      "kind": "cli",
      "command": "codex",
      "capabilities": ["code-review", "patch", "tests"]
    },
    {
      "id": "claude",
      "kind": "cli",
      "command": "claude",
      "capabilities": ["implementation", "product-review", "patch"]
    }
  ]
}
```

启动前检查：

- 命令是否存在。
- 当前工作区是否可信。
- 是否允许该 provider 获取工作区上下文。
- 是否需要用户登录或已有会话。

### 7.2 通过 ttmux session 管理 Agent

每个 review job 一个 reviewer session：

```text
review-<swarm>-<member>-codex
review-<swarm>-<member>-claude
```

好处：

- 输出可 capture。
- 人类可 attach 查看。
- 可以追加指令。
- 崩溃或卡住可 kill。
- 与现有 swarm/session 体系一致。

### 7.3 Prompt 模板

Reviewer prompt 必须强约束输出：

```text
你是本任务的代码审查 Agent。

目标：
- 只审查本次变更范围。
- 优先找真实 bug、回归、安全风险、测试缺口。
- 每个 finding 必须引用文件和行号，不能只给泛泛建议。
- 不要直接修改代码，除非后续被明确要求修复。
- 输出必须是 JSON findings 数组。

上下文：
- 用户目标：{{goal}}
- 作者 Agent：{{authorAgent}}
- 变更范围：{{diffSummary}}
- 相关文件：{{files}}
- 质量门禁：{{qualityGate}}
```

Author response prompt：

```text
你是这批改动的作者。

请逐条回应 reviewer finding：
- accept：承认问题，并说明修复计划。
- reject：必须引用代码或测试证据。
- clarify：说明需要人类决策的问题。

禁止为回避 finding 而做无关重构。
```

Judge prompt：

```text
你是裁决 Agent。

你只判断 reviewer finding 和 author response 哪一方证据更充分。
不要提出新需求。
每条输出 decision: accept / reject / needs-human。
```

## 9. 质量门禁策略

互审插件不应该只相信 Agent 文本，应结合可执行证据。

默认门禁：

```text
1. git diff 可解析
2. 项目 quick quality gate 可运行
3. 高危 finding 不得 open
4. accepted finding 必须有 fix commit/diff
5. verified finding 必须有复验记录
```

本仓库默认命令：

```bash
scripts/dev/quality/check.sh quick
```

PR 或运行时行为变更再跑：

```bash
scripts/dev/quality/check.sh full
```

插件可以把门禁写成 policy：

```json
{
  "qualityGate": {
    "quick": "scripts/dev/quality/check.sh quick",
    "full": "scripts/dev/quality/check.sh full",
    "blockOn": ["critical", "high"],
    "requireVerificationFor": ["critical", "high", "medium"]
  }
}
```

## 10. 安全与失控控制

### 10.1 防止 Agent 互相无限循环

限制：

- 每个 finding 最多三轮：review -> response -> judge。
- 每个 review job 最大时长，例如 20 分钟。
- 每个 swarm 同时最多 N 个 reviewer。
- reviewer 不能直接 spawn 新 reviewer，必须通过 Leader 或插件调度器。

### 10.2 防止无关改动

Fixer 修复 finding 时：

- 只能修改 finding 涉及文件，除非声明额外影响范围。
- 修复前后 diff 单独记录。
- 复验失败自动 reopen。

### 10.3 防止伪证据

要求：

- Finding 必须有文件/行号或命令输出。
- 命令输出必须来自插件执行记录，不接受 Agent 自称跑过。
- 引用的文件版本要记录 git hash 或 diff base。

## 11. 与通用插件机制的关系

通用插件机制是底座；智能互审插件是第一个“杀手级插件”。

它会反向验证底座是否够用：

| 底座能力 | Review Mesh 如何使用 |
|---|---|
| commands | 发起 review、复审、生成报告 |
| swarmHooks | 成员完成后自动审查 |
| agentTools | Agent 创建/处理 finding |
| views | Web 上展示互审面板 |
| permissions | 控制 git、测试、tmux、workspace 权限 |
| audit | 记录每条 finding、命令、裁决 |
| plugin host | 运行 review 调度器 |

因此插件机制 MVP 可以直接围绕 Review Mesh 建，而不是先做泛化市场。

## 12. MVP

第一版建议只做 Codex review Claude 产出的闭环：

1. `ttmux review start --target HEAD --by codex`
2. 插件收集 `git diff --stat`、`git diff`、用户目标。
3. 启动 Codex reviewer session。
4. 要求 Codex 输出结构化 findings。
5. findings 写入 `~/.local/share/ttmux/reviews/findings/`。
6. CLI 可 `ttmux review ls` 查看。
7. Web swarm 详情页后续再接。

MVP 不做：

- 插件市场。
- 多轮辩论。
- 自动修复。
- iframe UI。
- 组织 policy。

MVP 成功标准：

- 能稳定从一次真实 diff 里产出结构化 findings。
- 能把高危 finding 阻塞 swarm 集成。
- 人类能 attach reviewer session 看完整审查过程。
- finding 可以被手动标记 fixed/verified。
