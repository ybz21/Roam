# 客户端 / 服务端横向扩展设计

> 状态：设计草案（v0.3）
> 作者：cluster 方向
> 关联：本文是 [多节点跨网络管理设计.md](./多节点跨网络管理设计.md)（v0.1）的**实现落地版**。
> v0.1 用 Hub/Agent 描述概念，本文用用户口径的 **标准节点 / 云端 Broker**，并把每个
> 决策钉到 `backend/` 现有代码上，给出可直接开工的改动落点。
>
> **心智模型 = FRP**：云端 Broker ≈ `frps`（唯一公网入口、只转发不跑业务），
> 标准节点 ≈ `frpc`（在 NAT 后主动拨号注册进来）。区别是把这套内建成产品能力，
> 用户不用手配隧道，且能在**一个界面里切换**所有注册进来的节点。

---

## 0. 一分钟看懂

- 现在 Roam 是**单机**：一台机器跑一个 Go 后端（`backend/`，Gin 绑 `0.0.0.0:13579`），
  浏览器连它，所有能力（终端 / 文件 / git / browser / phone / swarm）都作用在**本机**。
- 目标：加一个**云端 Broker**，各开发机跑的**标准 Roam** 主动注册进来。用户在浏览器
  只登录**一个**云端地址，顶部切换"当前节点"，即可操控任意一台注册进来的机器——
  **被管理的机器不需要公网 IP、不需要开入站端口**。
- 就是 **FRP 的形状**：Broker = frps，标准节点 = frpc；但内建、免手配、带统一控制台。

一句话：**同一个二进制两种模式——标准（本机全套能力，可选注册进云端）、云端 Broker
（纯路由 + 控制台）；节点全部出站拨号连 Broker，Broker 只转发不搬业务。**

## 1. 术语对齐（避免和 v0.1 打架）

| 本文（用户口径） | v0.1 口径 | FRP 类比 | 是什么 |
| --- | --- | --- | --- |
| **云端 Broker**（Server） | Hub | `frps` | 有公网地址的中心节点。维护节点注册表、用户认证入口、把前端请求路由到目标节点、控制台入口。**不跑业务逻辑。** |
| **标准节点**（Client） | Agent | `frpc` | 每台开发机上的 Roam = 现有 `backend/` 全部能力。默认自管自；**可选**填云端地址后额外出站注册进云端。 |
| **Console** | Console | frps dashboard | 浏览器前端，跑在 Broker 上。顶部有"节点切换器"，切换所有注册进来的标准节点。 |

> 下文一律用 **标准节点 / 云端 Broker / Console**。"Client 端"= 注册进来的标准节点。

## 2. 核心设计决策

### 2.1 一个二进制，**两种**模式（角色由配置决定）

不做两个程序。`backend/` 编出来的同一个二进制，靠配置里的 `cluster.mode` 选角色：

| 模式 | 含义 | 谁跑业务 | 谁监听公网 | FRP 类比 |
| --- | --- | --- | --- | --- |
| `standard`（默认） | 现在的 Roam，跑全套能力。**可选**填 `cluster.broker` → 额外出站注册进云端 | 本机 | 本机（局域网直连口保留） | frpc（填了 broker 时） |
| `cloud` | 云端 Broker：纯路由 + 注册表 + 控制台 | 不跑（只转发） | 是（唯一公网入口） | frps |

关键简化 —— **没有单独的 "client 模式"**：一台开发机注册进云端后，它**本身还是
标准 Roam**，只是多连了一条到 Broker 的隧道。"Client" 只是"标准节点在云端视角下的
身份"，不是第三种部署形态。

**默认 `standard` 且不填 `broker`，现有用户零感知**——就是现在的单机行为。
这满足"Client 跟 Server 都部署成一种模式"的诉求：部署物永远是同一个二进制，
是否上云、当不当中心，只由配置决定。

落点：`backend/config/config.go` 现有 `type Web struct`（密码 / bind / tls…）。
新增 `type Cluster struct{ Mode, Broker, Token, Name, Group string }`，
`backend/cmd/main.go` 的 `main()`（现 line ~25-142）里按 `cfg.Cluster.Mode` 分两路。见 §10。

### 2.2 反向连接：节点主动拨号，Broker 永不回连

跨网络最大障碍：开发机通常在 NAT / 防火墙后，**没有公网 IP，无法被直接连接**。
所以**只有 Broker 需要公网地址**，标准节点全部**出站**连到 Broker，建一条**持久隧道**，
之后所有控制流量都在这条已建立的连接里双向复用。

> 所有箭头都是 **标准节点 → Broker 出站**、**浏览器 → Broker 出站**。
> 没有任何一方需要主动连到 NAT 后的机器。和 frpc→frps 完全同构。

### 2.3 Broker 是"路由器 + 注册表"，不重实现任何能力

Broker 只做四件事：**① 用户认证入口（复用 `backend/auth`）② 节点注册与心跳
③ 请求路由（把带 nodeId 的前端请求转发进目标节点的隧道）④ 接入签发（enrollment token）**。

每个标准节点复用**现有整套 Gin 路由**（`backend/server/server.go`），对现有 handler
**零改动**——等于把"本机 loopback"换成"隧道"。改动最小、风险最低。

### 2.4 复用项目已有的反代与 WS 技术栈

Broker 的转发不是新造轮子：
- HTTP/WS 反代直接照抄 `backend/browser/devtools.go` 的 `httputil.ReverseProxy` 用法
  （Go 1.12+ 的 `ReverseProxy` 原生支持 WebSocket 升级，项目已在 CDP 反代里验证过）。
- 隧道用 `gorilla/websocket`（`backend/go.mod` 已依赖 v1.5.3，终端 `backend/pty/pty.go`
  在用）承载，内部多路复用用 `yamux`。

## 3. 架构总览

```
                              ┌─────────────────────────────┐
   浏览器 (手机/平板/电脑)     │        云端 Broker          │  ≈ frps
   ┌──────────────┐           │   (公网, 唯一入口)           │
   │  Console     │  HTTPS    │  ┌───────────────────────┐  │
   │  节点切换器  │◄─────────►│  │ 用户认证 (密码+TOTP)  │  │  ← 复用 backend/auth
   └──────────────┘   wss     │  │ 节点注册表 / 心跳     │  │
                              │  │ 反向代理 / 路由        │  │  ← 照抄 browser/devtools.go
                              │  └──────────┬────────────┘  │
                              └─────────────┼───────────────┘
                    持久隧道 (出站, TLS+yamux)│  多路复用    ≈ frpc→frps
             ┌────────────────┬─────────────┴──────┬─────────────────┐
             ▼                ▼                    ▼                 ▼
      ┌────────────┐   ┌────────────┐       ┌────────────┐   ┌────────────┐
      │ 标准节点 A │   │ 标准节点 B │       │ 标准节点 C │   │ 标准节点 D │
      │ 家里台式机 │   │ 公司工作站 │       │ 云 GPU 机  │   │ 树莓派     │
      │ NAT 后     │   │ NAT 后     │       │ 有公网也无妨│   │ NAT 后     │
      │   = 现有 backend/ 全部能力 (term/files/git/browser/phone/swarm) + tunnel │
      └────────────┘   └────────────┘       └────────────┘   └────────────┘
```

部署形态：
- **推荐**：Broker 单独放一台便宜的公网小主机（职责干净，它不跑业务、资源占用小）。
- **也行**：Broker 和某台标准节点同机部署（那台机器既当中心又被管理）。
- **单机**：`mode: standard` 且不填 `broker`，和现在一模一样。

## 4. 统一二进制的启动分流

`backend/cmd/main.go` 按模式分两路，但 **cloud 不得复用当前 `server.New()`**。当前
`server.New()` 不只是拼路由，还会创建完整业务 handler、启动 `SyncLoop`，并初始化
browser / phone 等本地配置；先拆分装配边界，再选择 runtime：

```
main()
 ├─ 读 config（新增 cluster 段）
 ├─ mode == standard →
 │     · NewStandardRuntime(cfg) 建全套本地业务、后台循环与业务 Handler
 │     · NewConsoleServer(common, standard.Handler) 挂认证、静态资源和本地业务路由
 │     · 若 cluster.broker 非空：额外启 tunnel 客户端
 │           拨号 Broker → 注册 → 心跳 → 把隧道里的请求喂给本机同一个 Gin Handler
 │       若为空：就是现在的纯单机
 └─ mode == cloud →
       · NewBrokerRuntime(cfg) 只建注册表、隧道、授权与节点反代
       · NewConsoleServer(common, broker.Handler) 只挂认证、静态资源、Broker-local API
         和节点代理入口；绝不构造 StandardRuntime
```

建议把现有 `server.New()` 拆成 `NewCommonWeb`（cookie/session auth、前端静态资源、
fallback）、`NewStandardRuntime`（业务依赖、业务路由、SyncLoop、browser/phone 等）和
`NewBrokerRuntime`（注册表、隧道、Broker API、代理）。cloud 的依赖图中不允许出现
`StandardRuntime`，因此不会启动 SyncLoop、browser、phone、pty 或其他本地业务副作用。
标准节点上云与否仍复用同一个业务 `http.Handler`；上云只是让可信隧道 transport 将请求
投递给它。

新增包（建议）：
- `backend/cluster/broker/`：注册表、路由中间件、隧道服务端、enrollment。
- `backend/cluster/node/`：出站隧道、注册、心跳、把隧道请求灌进本机 `http.Handler`。
- `backend/cluster/tunnel/`：yamux over wss 的双向封装，Broker / 节点共用。

## 5. 节点如何配置 / 接入云端（用户视角）

一台标准节点要上云，只需告诉它 Broker 地址。两种入口，二选一：

### 5.1 一键接入（推荐）

Console 的 `/nodes` 页点「添加节点」→ Broker 生成**一次性 enrollment token** 和命令：

```bash
curl -fsSL https://<broker>/install.sh | bash -s -- \
    --broker https://<broker>:443 --token <ENROLLMENT_TOKEN> --name "公司工作站"
```

在开发机粘贴执行：装好 Roam（标准模式）、写好 `cluster.broker` 配置、用 token 注册到
Broker、换发长期节点凭证、建隧道。控制台节点列表里该机器**自动上线（绿灯）**。

### 5.2 手动配置

在开发机的 `~/.roam/config.yaml` 里，标准模式下补一段 `cluster`：

```yaml
cluster:
  mode: standard                   # 默认就是 standard，可省
  broker: https://<broker>:443     # 填了才上云；不填 = 纯单机
  token: <ENROLLMENT_TOKEN>        # 首次注册用；注册后自动换成长期凭证并回写
  name: 公司工作站                  # 节点显示名（可选，默认 hostname）
  group: 工作                       # 分组（可选）
```

启动即拨号接入。整个过程开发机**不需要公网 IP、不需要开入站端口、不需要手工配 frp**。

> 云端 Broker 侧配置：
> ```yaml
> cluster:
>   mode: cloud
>   # 复用 web.password / web.totp_secret 做用户登录；
>   # 复用 web.tls / web.bind 做公网入口
> ```

## 6. 认证与安全（三段独立信任链）

沿用 v0.1 的三段信任，钉到现有 `backend/auth`：

1. **用户 → Broker**：复用现有密码 + TOTP（`backend/auth/auth.go`），签发浏览器
   session cookie（`ttmux_session`，HttpOnly / SameSite=Strict / 7 天，现成能力）。
2. **节点 → Broker**：接入用**一次性 enrollment token**；首次注册后 Broker 换发
   **长期节点凭证**（长期 token / 节点私钥），落盘开发机 `~/.roam/node.json` 并回写
   config。隧道全程 TLS；进一步可上 **mTLS** 做节点身份强校验。
3. **浏览器 → 节点**：不直连。Broker 先校验用户 session 与目标节点授权，再通过已经
   完成节点双向认证的隧道投递。节点侧必须使用独立的内部 `http.Handler` 入口，或由
   tunnel `RoundTripper` 在进程内附加不可序列化的 transport context / typed principal；
   只有这个内部调用链可以绕过节点本地登录。本地监听端口始终进入 public handler。
   **禁止**依赖 `X-Forwarded-*`、`X-Roam-Internal` 等 HTTP header 判定“来自隧道”，
   因为公网或局域网客户端可以伪造 header。内部入口还要覆盖审计中的 Broker 用户与 nodeId。

补充约束：
- Broker 掌握"哪个用户能访问哪些节点 / 分组"的授权（为将来多用户预留，近期单用户）。
- 敏感操作（移除节点、签发接入 token）二次确认。
- 节点凭证与 `.env` 一样**不入库明文**，注册表里只存 `credentialRef`。

## 7. 通信与路由

### 7.1 隧道协议

- **载体**：WebSocket over TLS（走 443/wss，对企业防火墙最友好；复用 gorilla/ws）。
- **多路复用**：一条隧道内用 `yamux` 开多个逻辑流——一个终端会话、一次文件下载、
  一次 CDP 反代各占一条流，互不阻塞；心跳单独一条控制流。
- **断线重连**：节点侧指数退避重连；Broker 侧节点掉线即标 `offline`、拒绝路由
  （前端灰显该节点）。这和 `backend/pty` 里已有的重连 / 心跳思路一致。

### 7.2 前端如何指定目标节点

P1 统一采用稳定路径前缀 **`/n/<encodeURIComponent(nodeId)>/...`**。它既适用于封装请求，
也适用于浏览器原生发起、无法附加自定义 header 的资源请求；`nodeId` 是不可变内部身份，
显示名只用于渲染。Broker-local 与节点业务 API 必须分区：

- `/api/broker/*`：Broker 本地处理（节点列表、bootstrap、preferences、enrollment）；
  永不代理，也不需要 current node。
- `/n/<nodeId>/api/*`：节点业务 REST，包括 `api()`、普通 `fetch()` 和 multipart 上传。
- `/n/<nodeId>/api/term/...` 等：同一路径完成 WebSocket upgrade，终端、日志、browser stream
  均由共享 URL builder 生成。
- `/n/<nodeId>/api/files/...`、preview / image / HTML / Office / download 等：直接作为
  `<img src>`、`<iframe src>`、`<a href/download>` 或 `window.open()` 的原生资源 URL。

不能假设所有 REST 都经过 `frontend/src/api.ts`。实现前要盘点并改造 `api()`、散落的
`fetch()`、`FormData`/multipart、`FileView.tsx` 预览、`FileBrowser.tsx` 下载、
`PluginsPanel.tsx` 上传，以及所有 WS URL 拼接；统一收口为 `nodeApiUrl(nodeId, path)` 与
`nodeWsUrl(nodeId, path)`。路径中的 nodeId 必须逐段 percent-encode，Broker 解码后按精确
ID 查注册表，不接受显示名回退。兼容期可读取旧 `X-Roam-Node` / `?node=` 并 307 到规范
路径，但新代码不再产生它们。

Broker 从路径解析 nodeId → 校验当前用户授权 → 查注册表 → 转发进对应节点的隧道流。
转发本体就是 `httputil.ReverseProxy`，`Director` 里把目标换成"该节点隧道上的一个
虚拟 loopback"（用 yamux 流当 `http.RoundTripper` 的连接）。这套 Broker 已经在
`backend/browser/devtools.go` 用同款反代（含 WS 升级、改 Host、清 Origin）验证过。

> 二期可加**路径前缀** `/n/<nodeId>/api/...` 之外的分享直链（把 UI 状态编码进 URL），
> 便于分享"直链到某节点某会话"。

### 7.3 终端流（关键路径，验证零改动）

```
现在:  浏览器 wss /api/term/:name ──► 本机 pty ──► tmux attach          (backend/pty/pty.go)
之后:  浏览器 wss /n/B/api/term/:name ──► Broker ──► 隧道 ──► 节点 B 的
        /api/term/:name ──► B 本机 pty ──► tmux attach
```

对 pty 层（`backend/pty/pty.go` 的 upgrader + 双向 copy）**零改动**，只是外面套了
一层 Broker 转发。文件 / git / browser / phone / swarm 同理——数据源从"本机"变成
"当前选中节点"，handler 不动。

## 8. 数据模型（Broker 侧）

```
Node {
  id            string    // 稳定唯一 id
  name          string    // 用户可改的显示名
  hostname      string
  os            string    // linux/darwin/...
  version       string    // 节点的 Roam 版本
  capabilities  []string  // term/files/git/browser/phone/swarm（节点上报）
  group         string
  tags          []string
  status        enum      // online | offline
  lastHeartbeat time
  sessionCount  int       // 节点上报的会话数
  load          float
  credentialRef string    // 指向长期凭证（凭证本身不明文入库）
}

EnrollmentToken { token, expiresAt, usedAt, presetName, presetGroup }
```

持久化用轻量存储（bbolt / SQLite / JSON 文件均可，与项目现有习惯对齐；注册表读多写少）。

## 9. 前端改动（Console，跑在 Broker 上）

> 交互与逐页面细节（切换器放哪、切机器时各页行为、终端标签机器标识、离线防护、
> 深链方案）见专文 [多机切换交互设计.md](./多机切换交互设计.md)。本节只给要点。

- **节点切换器**（全局常驻，顶栏左上角）：`● 公司工作站 ▾`（圆点=在线灯，绿/黄延迟/灰离线）。
  点开下拉是节点列表（搜索 + 延迟 + 会话数 + 添加/管理入口）。
- 全局状态存 `currentNodeId`；共享 URL builder 为各种 HTTP / WS / 原生资源入口生成
  `/n/<nodeId>/...` 路径。
  切节点 = 换 `currentNodeId` + 刷新当前页数据，**无需换 URL / 重新登录**。
- **节点管理页 `/nodes`**：节点卡片网格（host / OS / 版本 / 在线 / 心跳 / 延迟 /
  会话 / 负载 / 分组），顶部「＋ 添加节点」弹出接入命令（§5.1）。
- 现有页面（终端 / 文件 / git / browser / phone / swarm）**功能不变**，只在顶部加一条
  "当前节点"面包屑，避免在 A 机器上误操作 B 机器。
- **i18n 硬约束**：节点切换器、`/nodes` 页、接入弹窗、状态标签等所有新文案必须走项目
  i18n 层（见 `docs/development/i18n.md`），不得新增硬编码中英文。

> 标准模式单机跑时，前端**不显示**节点切换器（没有云端就没有多节点概念），保持现状。

| Broker：路由中间件 | `backend/cluster/broker/router.go`（新增） | 解析 `/n/:nodeId/*`、鉴权后用 `httputil.ReverseProxy` 转发进隧道流（照抄 `backend/browser/devtools.go`） |
| Broker：隧道服务端 + enrollment | `backend/cluster/broker/tunnel.go`（新增） | 接受节点出站长连、校验 token、换发长期凭证 |
| Web/runtime 拆分 | `backend/server/` | 拆 `NewCommonWeb` / `NewStandardRuntime` / `NewBrokerRuntime`；cloud 不构造本地业务 runtime |
| Broker：装配路由 | `backend/server/` | `/api/broker/*` 本地处理；`/n/:nodeId/*` 代理；`/cluster/tunnel` 接入节点 |
| 节点：出站隧道 | `backend/cluster/node/node.go`（新增） | 拨号 / 注册 / 心跳 / 重连；仅通过内部 transport 调用 standard 业务 Handler |

## 11. 分阶段实施路线

| 阶段 | 内容 | 产出 |

## 12. 与现有 frp 方案的关系

本方案**就是把 frp 内建成产品**：Broker = frps、标准节点 = frpc。
- **frp 不废弃**：也可以先用 frp 把 Broker 的公网入口暴露出来，节点再朝这个入口拨号。
- 本方案取代的是"**每台机器各配一条 frp + 各自 URL / 密码**"这种散装用法，改为
  "一个入口、一套认证、内建隧道、页面里切换"。
- 没有公网机器的用户：Broker 放一台便宜公网小主机，节点全在 NAT 后拨入。

## 13. 待用户确认的关键取舍

1. **Broker 部署形态**：独立公网小主机（推荐，干净）还是复用某台标准节点同机部署？
2. **隧道多路复用库**：`yamux` vs `smux`（都行，倾向 yamux，社区更广）。
3. **节点标识传递**：已定为稳定 nodeId 的路径前缀 `/n/<nodeId>/...`（覆盖 fetch/multipart/
   img/iframe/download/window.open/WS 等所有入口，不依赖 header 注入）；显示名只渲染、不参与
   路由；`/api/broker/*` 为 Broker-local。详见 §7.2。旧 `X-Roam-Node`/`?node=` 仅兼容迁移。
4. **多用户 / 授权**是否纳入近期范围，还是先单用户管多机、团队版后置。
5. **注册表持久化**：bbolt / SQLite / JSON 文件——按项目现有习惯定。
