# 多机切换：交互与页面设计

> 状态：设计草案（v0.1）
> 关联：架构见 [客户端-服务端横向扩展设计.md](./客户端-服务端横向扩展设计.md)（两模式：标准节点 / 云端 Broker）。
> 本文只管**前端交互和页面**——加了多台机器之后，用户在浏览器里怎么看、怎么切、
> 每个页面变了什么。**核心原则：最大化沿用现有三端布局，不另起一套。**
>
> 📐 **静态页面原型**（每页一个 HTML，浏览器直接打开）：[mockups/index.html](./mockups/index.html)
> ——概览/项目/文件/浏览器/手机/机器/插件/设置各一页 + 手机端合集；左侧栏对齐现有实现、无 emoji，
> 机器切换器选中的当前机器跨页保持。

---

## 0. 一分钟看懂

- 单机用户（标准模式没连 Broker）：**界面零变化**，看不到任何多机元素。
- 连了 Broker 之后：顶部（桌面在左侧栏顶、手机在细顶条）多一个**机器切换器**
  `● 公司工作站 ▾`。切一下，浏览类页面（概览 / 会话 / 文件 / git / browser / phone）
  整体换到那台机器。
- **已经打开的终端标签不会因为切机器而消失**——它们自带机器标记（`公司工作站·build`），
  是你正在跑的活；切机器只影响"新打开 / 正在浏览"的目标。

---

## 1. 设计原则

1. **沿用现有骨架**：桌面左侧 `Layout.Sider`（App.tsx:462）、手机底部 Tab 栏
   （App.tsx:602）、终端标签条（App.tsx:964）、`NAV` 导航表（App.tsx:51）——
   全部复用，机器切换器**挂进这些既有位置**，不新增布局层。
2. **单机零感知**：没连 Broker 时，切换器整个不渲染，和今天像素级一致。
3. **一眼知道"我在哪台机器"**：切换器常驻可见 + 终端头部显机器名，杜绝"在 A 上
   误删了 B 的东西"。
4. **不丢工作**：切机器不关任何已打开的终端。终端是长活上下文，跟机器绑定、能并存。

---

## 2. 心智模型（最关键的取舍）

问题：多台机器后，"当前机器"到底是**全局模式**（选中 B，一切都显示 B），还是
**每个资源各自带机器**（这个终端在 B、那个文件页在 A）？

**结论：混合。**

| 页面类型 | 归属 | 说明 |
| --- | --- | --- |
| **浏览类**：概览 / 会话列表 / 文件 / git / browser / phone | 跟**全局「当前机器」** | 切换器一变，这些页整体 re-scope 到新机器并刷新 |
| **终端标签**（已打开的） | **各自带机器标记，常驻并存** | 复用现有多标签条；`公司工作站·build`、`GPU机·train` 可同时开着，切机器不关它们 |

**为什么终端要特殊对待**：现有 App 本来就支持多终端标签并存（App.tsx:964 的 tab 条）。
如果切机器就把别台的终端关掉，用户"瞄一眼另一台机器的状态"就会丢掉正在跑的
Claude/训练会话——这在实际使用里不可接受。所以：
- **切机器** = 换"我正在浏览 / 新打开操作会落到哪台机器"（浏览类页面 + 新建的默认目标）。
- **终端标签** = 打开那一刻就钉死在某台机器上，带机器前缀，跨切换保留。

> 备选（未采纳）：纯全局模式——切机器关掉所有别台终端。实现更简单，但丢工作，pass。
> 备选（未采纳）：纯 per-page——每个页面各记各的机器。心智负担重，"我现在在哪台"
> 变模糊，pass。

## 4. 终端标签的机器标识（复用现有标签条）

现有标签条（App.tsx:964-982）今天显示 `build`、`deploy`。多机后加**机器前缀 + 颜色点**：

```
现在:  [● build] [● deploy]
之后:  [🟦 公司·build] [🟩 GPU·train] [🟦 公司·deploy]
        └─ 蓝=公司工作站  └─ 绿=GPU机（每台机器一个稳定色点，一眼分组）
```

- 每台机器分配一个稳定色（hash nodeId → 颜色），标签左侧色点 = 机器身份。
- 标签文字 `<机器短名>·<会话名>` **仅用于显示**；寻址与深链一律用稳定 `nodeId`
  （标签态编码为 base64url(JSON)，见 §9），显示名可重命名/重复也不影响。
  空间紧时机器名收成色点+首字，hover 出全名。
- **切机器不影响这些标签**：它们各自钉在自己机器上。点某个别台机器的标签，
  可选择"顺便把当前机器也切过去"（弱联动，见 §5）。
- 终端头部（App.tsx:983-1033 / 1158-1163 的状态条）显示完整 `公司工作站 · build ● 已连接`。

移动端终端是全屏页，头部同样显机器名，避免全屏时忘了这是哪台机器。

## 5. 切换时各页面的行为

切换器选中机器 B：

1. 全局 `currentNodeId = B`（bootstrap / 持久化见 §9）。
2. **浏览类页面**（概览 / 会话 / 文件 / git / browser / phone）：清空当前数据、
   改用 `/n/B/api/...` 重新拉取。文件页回到 B 的家目录、git 页回到 B 的仓库列表。
3. **终端标签**：原样保留，不关不切。
4. 顶部切换器与页面面包屑同步显示 B。

**弱联动（已定：要）**：点标签条里某个属于机器 C 的终端 → 弹一个小提示气泡
「切到 云GPU机 一起看？」，点了就把 `currentNodeId` 也切到 C（浏览类页面跟着走）；
不点就只是聚焦那个终端、浏览上下文不动。避免"看着 C 的终端、旁边文件页却是 B"的错位，
又不强制。（设置里可关，默认开。）

## 8. 离线、重连、误操作防护

- **离线机器**：切换器里灰显、不可选（复用现有 disabled 态）。
- **当前机器掉线**：页面顶部浮一条 antd `Alert`「公司工作站 已离线，正在重连…」，
  期间浏览类页面数据置灰、写操作禁用；该机器的终端标签显"连接断开"并自动重连
  （复用 App.tsx 现有断线重连提示）。
- **误操作防护**：每个浏览类页面顶部一条**当前机器面包屑**（`公司工作站 / 文件 / …`），
  终端头部显机器名——始终提醒"你在操作哪台机器"。

## 9. URL / 深链方案（沿用现有 hash + query）

现有是 hash 路由 + query 参数（`#/files`、`?terms=…&active=…`，App.tsx:236-244）。
把机器织进去，**不换路由框架**，但身份只使用稳定 `nodeId`：

- 浏览类：`#/files?nodeId=<percent-encoded-node-id>`。显示名可重命名或重复，绝不参与寻址。
- 终端不再使用 `公司·build,GPU·train` 这种脆弱拼接。`terms` / `active` 的值使用
  base64url(JSON) 编码，例如 `terms=<b64url([{"nodeId":"n_01","term":"build"}])>`、
  `active=<b64url({"nodeId":"n_01","term":"build"})>`；JSON 字符串可无歧义承载逗号、
  中点、Unicode 和任意会话名。
- 兼容解析：若存在旧 `node=`，仅当它精确匹配唯一节点 ID 时采用；旧显示名仅可在
  唯一匹配时一次性迁移并立刻 `replaceState` 写回 `nodeId=`，重复或不存在则回退 bootstrap。
  旧 `terms=公司·build,...` 只做尽力迁移（按最后一个中点拆分并要求显示名唯一），
  有歧义的条目丢弃并给本地化提示；新链接永不生成旧格式。

### 9.1 bootstrap 与持久化（消除启动循环）

`currentNodeId` 不放进节点侧、需要 current node 才能读取的 preferences API。启动顺序为：

1. 先读 URL `nodeId`，其次读 `localStorage['roam.currentNodeId']`；这两者都不发网络请求。
2. 无候选或候选失效时请求 Broker-local `GET /api/broker/bootstrap`，返回用户可访问节点、
   Broker 级偏好和推荐节点；该 namespace 永远在 Broker 本地处理，不依赖 current node。
3. 选中后写 localStorage，随后才允许发 `/n/<nodeId>/...` 节点业务请求。
4. 若需要跨浏览器同步，可异步 `PUT /api/broker/preferences` 保存 `currentNodeId`；失败不阻塞
   启动，本地值仍可用。节点自身 `/n/<nodeId>/api/preferences` 只保存该节点业务偏好。

## 11. 分阶段（和架构文档的路线对齐）

- **P1**：切换器（桌面 Sider 顶 + 手机细条）+ 终端标签机器标识 + 共享 node URL builder
  （`nodeApiUrl`/`nodeWsUrl` 生成 `/n/<nodeId>/...`）+ Broker bootstrap（§9.1）+ `/nodes` 基础页。
  **不是靠 header 自动注入**——需逐类盘点改造请求入口：`api()`、散落 `fetch()`、multipart、
  预览/下载、img/iframe/window.open、全部 WS（见 §10）。
- **P2**：弱联动、离线 Alert/重连、深链恢复带机器、机器色点/分组、接入命令一键复制。
- **P3**：跨机器总览 Dashboard（一屏看全部机器）、跨机器 swarm 分派可视化。
