# 多节点跨网络管理设计

> 状态：设计草案（v0.1）— 概念版
> 实现落地版见 [客户端-服务端横向扩展设计.md](./客户端-服务端横向扩展设计.md)（v0.3，
> 两模式模型：标准节点 / 云端 Broker（≈ frpc / frps），钉到 `backend/` 现有代码，给出改动落点）。
> 目标：把 Roam 从"单机 Web 遥控台"扩展为"一个控制台管理多台开发机"，
> 且这些机器可以分布在不同网络（家里、公司、云、NAT/防火墙后面）。

---

## 1. 背景与现状

当前 Roam 是**单机**架构：一台开发机上跑一个 Go 后端（`backend/`，Gin +
`0.0.0.0:13579`），前端（React + xterm.js）连它，所有能力（tmux/pty、文件、
git、browser 镜像、phone 镜像、swarm）都作用在**本机**。

跨网络访问目前靠**运维手段**（`docs/deploy/frp.md`）：一台一条 frp 隧道，
每台机器一个独立 URL、独立登录。机器一多，这套就散了：

- 每台机器各自一个地址、各自密码，切换靠"换浏览器标签页"；
- 没有统一的"这些机器现在都在干什么"的总览；
- NAT 后的机器要手工配 frp，接入成本高；
- 无法把一个任务/蜂群跨机器编排。

用户诉求：**要有一个中心节点，其他机器都连进来，用一套控制台管理多台机器并在它们之间切换。**

---

## 2. 核心设计决策

### 2.1 反向连接（Agent 主动拨号 Hub）

跨网络最大的难点是：被管理的机器通常在 NAT / 防火墙后，**没有公网 IP，
无法被中心直接连接**。

因此采用**反向连接**：只有中心节点（Hub）需要一个公网可达地址，各被管理机器
上的 Agent **主动出站**连到 Hub，建立一条**持久隧道**。之后所有控制流量都在这条
已建立的隧道里双向复用。这与 frp 的思路一致，但把它**内建成产品能力**，用户不
再手工配隧道——在目标机器跑一条接入命令即可。

> 一句话：**Hub 只需一个公网入口；Agent 全部朝 Hub 拨号，永不需要公网 IP。**

### 2.2 Hub 是"路由器 + 注册表"，不搬运业务逻辑

Hub 不重新实现 tmux/文件/git 等能力，它只做三件事：**节点注册与心跳、
请求路由（把前端请求转发到目标 Agent）、统一认证入口**。每个 Agent 复用**现有
整套 Roam 后端**，只多加一个"隧道客户端"。这样改动最小、风险最低。

### 2.3 渐进式，不推倒重来

Agent = 现有 `backend/` + 一个 outbound 隧道模块。Hub 复用现有技术栈（Go/Gin/
`httputil.ReverseProxy`，项目已在 browser CDP 反代里用过）。单机模式继续可用
（本机既是 Hub 又是唯一 Agent）。

## 3. 架构总览

三个角色：

- **Hub（中心节点 / 控制面）**：一台有公网地址的机器（云主机，或用户自己有公网
  的机器）。维护节点注册表、路由表、用户认证、Web 控制台入口。**不跑业务逻辑。**
- **Agent（被管理节点）**：每台开发机上运行 = 现有 Roam 后端 + 隧道客户端。
  主动连 Hub，上报心跳/能力/状态，接收并执行来自 Hub 转发的请求。
- **Console（Web 前端）**：浏览器连 Hub。顶部有"节点切换器"，选中某节点后，
  所有操作都作用在该节点上。

```
                            ┌─────────────────────────────┐
   浏览器 (手机/平板/电脑)   │            Hub               │
   ┌──────────────┐         │   (公网, 唯一入口)           │
   │  Console     │ HTTPS   │  ┌───────────────────────┐  │
   │  节点切换器  │◄───────►│  │ 用户认证 (密码+TOTP)  │  │
   └──────────────┘  wss    │  │ 节点注册表 / 心跳     │  │
                            │  │ 反向代理 / 路由        │  │
                            │  └──────────┬────────────┘  │
                            └─────────────┼───────────────┘
                        持久隧道(出站, TLS)│  多路复用
             ┌────────────────┬───────────┴────────┬─────────────────┐
             ▼                ▼                    ▼                 ▼
      ┌────────────┐   ┌────────────┐       ┌────────────┐   ┌────────────┐
      │  Agent A   │   │  Agent B   │       │  Agent C   │   │  Agent D   │
      │ 家里台式机 │   │ 公司工作站 │       │ 云 GPU 机  │   │ 树莓派     │
      │ NAT 后     │   │ NAT 后     │       │ 有公网也无妨│   │ NAT 后     │
      │            │   │            │       │            │   │            │
      │ = 现有 Roam后端 + 隧道客户端（tmux/文件/git/browser/phone/swarm）│
      └────────────┘   └────────────┘       └────────────┘   └────────────┘
```

关键点：所有箭头方向都是 **Agent → Hub 出站**，以及 **浏览器 → Hub 出站**。
没有任何一方需要主动连到 NAT 后的机器。

## 4. 组件设计

### 4.1 Hub

| 子模块 | 职责 |
| --- | --- |
| 用户认证 | 复用现有 `backend/auth`（密码 + TOTP），签发面向浏览器的 session token |
| 节点注册表 | 记录每个节点：id、名称、主机名、OS、版本、能力、在线状态、最后心跳、当前会话数、分组、标签 |
| 隧道服务端 | 接受 Agent 出站长连接，校验节点凭证，维护 `nodeId → 隧道` 映射 |
| 路由/反代 | 把带 `nodeId` 的前端请求（REST + WebSocket）转发进对应节点的隧道 |
| 接入签发 | 生成一次性 enrollment token 与接入命令，供新机器加入 |

Hub 可与某台 Agent **同机部署**（那台机器既当中心又被管理），也可单独部署在一台
瘦公网小主机上（推荐，职责清晰）。

### 4.2 Agent

= 现有 `backend/` 全部能力 **+** 一个新模块 `backend/tunnel/`（出站隧道客户端）：

- 启动时用节点凭证连 Hub，建立持久隧道（断线自动重连，指数退避）；
- 定期心跳，上报状态（会话数、负载、版本、能力清单）；
- 把从隧道收到的 HTTP/WebSocket 请求，交给**本机现有的 Gin 路由**处理，再把
  响应写回隧道——对现有 handler **零改动**，等于把"本机 loopback"换成"隧道"。

Agent 仍可独立监听本地端口（局域网直连 / 兼容 frp），隧道只是"多一条到 Hub 的路"。

### 4.3 Console（前端）

- 顶部/侧栏新增**节点切换器**：当前节点名 + 在线状态灯 + 延迟；点开是节点列表。
- 全局状态里保存稳定 `currentNodeId`；所有节点请求使用 `/n/<nodeId>/...` 路径，
  Broker-local API 使用 `/api/broker/*`（见 §5）。切换节点无需重新登录。
- 新增**节点管理页** `/nodes`（见 §8）。

## 5. 通信与路由

### 5.1 隧道协议

- **载体**：WebSocket over TLS（复用现有 gin + gorilla/ws 技术栈；走 443/wss 对
  企业防火墙最友好）。
- **多路复用**：一条隧道内用 `yamux`/`smux` 开多个逻辑流，一个终端会话、一次文件
  下载、一次 CDP 反代各占一条流，互不阻塞。心跳单独一条控制流。
- **断线重连**：Agent 侧指数退避重连；Hub 侧在节点掉线时把其标记为 offline 并
  拒绝路由（前端灰显该节点）。

### 5.2 前端如何指定目标节点

统一使用路径前缀 `/n/<percent-encoded-nodeId>/...`。它同时覆盖封装 REST、普通 fetch、
multipart、WebSocket，以及不能加自定义 header 的 img / iframe / download 原生请求。
`/api/broker/*` 始终由 Hub 本地处理，因此 bootstrap 和偏好读取不依赖当前节点。
共享 URL builder 负责所有路径拼接；显示名不参与寻址。旧 header/query 只作兼容迁移。

Hub 转发逻辑基本就是 `httputil.ReverseProxy`，`Director` 里把目标换成"该节点隧道
的一个虚拟 loopback"。项目已在 `backend/browser/devtools.go` 用同款反代（含
WebSocket 升级），可直接借鉴。

### 5.3 终端流（关键路径）

现有：浏览器 `wss /api/term/:name` → 本机 pty → `tmux attach`。
改造后：浏览器 `wss /n/B/api/term/:name` → **Hub** → 隧道 → **Agent B** 的
`/api/term/:name` → B 本机 pty。对 pty 层**零改动**，只是外面套了一层转发。

## 6. 认证与安全

三段信任链，逐段独立：

1. **用户 → Hub**：复用现有密码 + TOTP，签发浏览器 session token（现成能力）。
2. **Agent → Hub**：节点接入用**一次性 enrollment token**；首次注册后 Hub 给该
   节点换发**长期节点凭证**（节点私钥 / 长期 token，落盘 `~/.local/share/ttmux/node.json`）。
   隧道全程 TLS；进一步可上**双向 TLS**（mTLS）做节点身份强校验。
3. **浏览器 → Agent**：不直连。Agent 通过独立 internal handler 或进程内 typed
   transport principal 识别已认证隧道调用；公网监听只进 public handler。不得以任何
   可由客户端伪造的 HTTP header 作为绕过本地鉴权的依据。

补充：
- Hub 掌握"哪个用户能访问哪些节点/分组"的授权（为将来多用户预留）；
- 敏感操作（移除节点、签发接入 token）需二次确认；
- 隧道凭证与 `.env` 一样**不入库**。

## 7. 节点接入流程（用户视角）

1. 控制台 `/nodes` 页点「添加节点」→ Hub 生成一次性 enrollment token 和一条命令：
   ```bash
   curl -fsSL https://<hub>/install.sh | bash -s -- \
       --hub https://<hub>:443 --token <ENROLLMENT_TOKEN> --name "公司工作站"
   ```
2. 在目标机器粘贴执行：装好 Agent、写入节点凭证、注册到 Hub、建立隧道。
3. 控制台节点列表里该机器**自动上线**（绿灯），即可切过去用。

整个过程目标机器**不需要公网 IP、不需要开入站端口、不需要手工配 frp**。

## 8. 页面设计

### 8.1 节点切换器（全局，常驻）

顶栏左上角组件：`● 公司工作站  ▾`（圆点=在线状态，绿/黄延迟/灰离线）。点开下拉：

```
┌─────────────────────────────┐
│ 🔍 搜索节点…                │
├─────────────────────────────┤
│ ● 公司工作站   12ms   3 会话 │  ← 当前
│ ● 家里台式机   45ms   1 会话 │
│ ● 云 GPU 机    88ms   0 会话 │
│ ○ 树莓派       离线          │
├─────────────────────────────┤
│ ＋ 添加节点   ⚙ 管理节点     │
└─────────────────────────────┘
```

选中即切换当前上下文，会话/文件/git/browser/phone 全部跟着切到该节点。

### 8.2 节点管理页 `/nodes`

节点卡片网格，每张卡片：

```
┌──────────────────────────────────┐
│ ● 公司工作站            [进入]    │
│ host: dev-ws-01 · Linux · v1.4.2  │
│ 在线 · 心跳 2s 前 · 延迟 12ms     │
│ 会话 3 · 负载 0.7 · 分组: 工作    │
│ [重命名] [分组] [详情] [移除]     │
└──────────────────────────────────┘
```

顶部「＋ 添加节点」弹窗给出接入命令（§7）。支持按分组/标签过滤、按在线状态排序。

### 8.3 全局总览 Dashboard（可选，二期）

一屏看全部节点：在线数、总会话数、各节点跑着的 agent/swarm、异常告警。
点任意条目直接跳到对应节点对应会话。

### 8.4 现有页面的改动

终端、文件、git、browser、phone、swarm 页**功能不变**，只是数据源从"本机"变为
"当前选中节点"。顶部显示"当前节点"面包屑，避免在 A 机器上误操作 B 机器。

## 9. 数据模型（Hub 侧）

```
Node {
  id            string   // 稳定唯一 id
  name          string   // 用户可改的显示名
  hostname      string
  os            string   // linux/darwin/...
  version       string   // Agent 版本
  capabilities  []string // term/files/git/browser/phone/swarm
  group         string
  tags          []string
  status        enum     // online | offline
  lastHeartbeat time
  sessionCount  int
  load          float
  credentialRef string   // 指向节点长期凭证（凭证本身不明文入库）
}

EnrollmentToken { token, expiresAt, usedAt, presetName, presetGroup }
```

持久化用轻量存储（bbolt / SQLite / JSON 文件均可，与项目现有习惯对齐）。

## 10. 分阶段实施路线

| 阶段 | 内容 | 产出 |
| --- | --- | --- |
| **P0** | 现状盘点 + 本设计定稿 | 本文档 |
| **P1 · 打通** | Hub 反代骨架 + Agent 出站隧道 + 节点注册/心跳 + 前端节点切换器 + `/nodes` 基础页 | 能加多台机器、能切换、能用终端 |
| **P2 · 好用** | 断线重连、节点分组/标签、延迟/负载展示、直链到节点会话、接入命令一键复制 | 日常可用的多机控制台 |
| **P3 · 编排** | 跨节点 swarm（把 swarm 成员分派到不同机器）、全局 Dashboard、告警 | 跨机器协同 |
| **P4 · 治理** | 多用户 + 按节点/分组授权、mTLS、审计日志 | 团队/生产级 |

先做 **P1** 验证"反向隧道 + 路由 + 切换"主链路，再逐步加料。

## 11. 与现有 frp 方案的关系

- **frp 不废弃**：它仍是"把单台 Hub 暴露到公网"的一种简单选择——即用 frp 暴露
  Hub 的公网入口，Agent 再朝这个入口拨号。
- 本方案取代的是"**每台机器各配一条 frp + 各自 URL/密码**"这种散装用法，改为
  "一个入口、一套认证、内建隧道、页面里切换"。
- 对没有公网机器的用户：Hub 放一台便宜公网小主机 + frp/直接绑公网，Agent 全部
  在 NAT 后拨入即可。

## 12. 待用户确认的关键取舍

1. **Hub 部署形态**：独立公网小主机（推荐，干净）还是复用某台开发机同机部署？
2. **隧道多路复用库**：`yamux` vs `smux`（都行，倾向 yamux，社区更广）。
3. **节点标识传递**：已定为稳定 nodeId 路径前缀；显示名只渲染，不参与路由。
4. **多用户/授权**是否纳入近期范围，还是先单用户管多机、后面再做团队版。
