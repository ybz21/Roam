# 蜂群成员启动与提示词模板设计

> 前置：建立在 [蜂群编排设计.md](蜂群编排设计.md)（swarm/member/master/依赖门控）与
> [蜂群广场与看板设计.md](蜂群广场与看板设计.md)（Plaza/Board）之上。
>
> 状态：v0.2 —— **M1–M4 已实现**（kind/role 列 + CLI `--kind/--role` + codex 引擎 +
> 后端 text/template 渲染 + 前端引擎选择/master tag）。M5（master 收敛）按现有 `swarm new`
> 的 master 开关达成（关掉即「首个成员=master」的 option A），未改默认行为。
>
> 实现备注：
> - 后端把渲染好的整段 prompt 作为 `task` 传给 `swarm add`（`prompt_rendered` 直接落 `task` 列）。
>   故 `status --json` 用 `\x1e` 记录分隔 + `read -d` 读，避免多行 task 撑破 JSON。
> - **蜂群 agent 成员走交互式常驻** claude/codex（`AGENT_INTERACTIVE=1`，prompt 作首条消息实参注入，
>   `%q` 安全转义）——可从前端终端**进入/接管**，完成靠 `swarm done`，不靠进程退出（对齐"成员长驻"本意）。
>   `ttmux spawn --agent`(一次性并行)仍走 `claude -p`。
> - 拓扑图 master 顶点用 `role=master` 成员；无成员 master 且无 supervisor 时不再画幽灵 `cc-<群>` 节点。

## 1. 背景与目标

当前「加成员」启动的 agent **只有 claude**，且喂给它的 prompt 就是用户填的一行任务，
agent 并不知道自己在一个蜂群里、该怎么和同伴协作、有哪些 skill 可用。本设计补四件事：

1. **可选引擎**：加成员时默认起 `claude`，可切换 `codex`。
2. **提示词模板**：成员的 prompt 由模板渲染——告诉它蜂群上下文、`skills/` 可用、走 swarm 协作格式。
3. **角色**：一个蜂群里**第一个**成员默认是 **master**（带 tag），其余是 **worker**；两种角色用不同模板。
4. **模板引擎选型**：后端是 Go，给出结论（见 §6.1）。

## 2. 现状（代码事实）

| 事实 | 位置 |
|------|------|
| agent 成员启动命令：`cd dir && claude -p … <<'EOF' {task} EOF` | `cli/ttmux-cli/lib/agent.sh:16` `_agent_claude_cmd` |
| **任务文本 = claude 的 prompt**（heredoc body） | 同上 |
| 仅支持 claude（`AGENT_CLAUDE_BIN`），**无 codex 启动** | `cli/ttmux-cli/lib/agent.sh:7` |
| `swarm add <群> <成员> --type task\|agent [--dir/--perm/--model] [--depends-on] <任务>` | `cli/ttmux-cli/lib/swarm.sh:228` |
| members 表：`name,type,task,workdir,status,deps,done,pending,model,perm` | `cli/ttmux-cli/lib/store.sh:72` |
| master 是 `swarm new` 单独拉起的 `cc-<群>` 会话（supervisor），**非成员** | `cli/ttmux-cli/lib/swarm.sh:216` |
| 后端 `SwarmAddMember` 薄透传 `{name,type,task,dir,deps,model,perm}` → CLI | `backend/api/swarm.go:38` |
| 前端加成员表单：Segmented[Agent(Claude)\|命令] + 名/任务/目录/依赖 | `frontend/src/Swarm.tsx:265` |
| 可用 skill：`skills/cc-swarm/`（蜂群协作）、`skills/ttmux/`；装到 `~/.claude/skills/` | `install.sh` |

**正交属性厘清**（设计后）：`type`=执行类型(task=shell / agent=AI)，`kind`=引擎(claude/codex)，`role`=角色(master/worker)。三者独立；`type=task` 时 kind/role 不适用。

## 3. 数据模型变更

members 表加两列（向后兼容，带默认值，旧库 `ALTER TABLE … ADD COLUMN` 平滑迁移）：

```sql
ALTER TABLE members ADD COLUMN kind TEXT DEFAULT 'claude';  -- 引擎: claude | codex
ALTER TABLE members ADD COLUMN role TEXT DEFAULT 'worker';  -- 角色: master | worker
```

> 迁移：`store.sh` 的 schema 初始化里用 `_col_exists` 守卫补列（`PRAGMA table_info(members)`），
> 避免对已有蜂群库报错。`prompt`（最终渲染的提示词）**不入库**——库存 `task`(原始任务)即可，
> prompt 是渲染产物，按需重算；若要可复现，可另存 `prompt_rendered TEXT`（可选）。

## 4. 角色：master / worker

### 规则
- 加 **agent** 成员时：若该蜂群**还没有 master 成员** → 本成员默认 `role=master`；否则 `worker`。
- UI 上 master 带醒目 tag（拓扑图节点已有 master 视觉，复用 `nodeIcon` ◆ / magenta）。
- 用户可在表单里手动改角色（高级）。

### 与现有 `cc-<群>` master 的调和（**需拍板**）
现状 master 是 `swarm new` 拉起的独立 cc 会话。两条路线：

- **A（推荐）· master 收敛为成员**：不再自动拉 `cc-<群>`；第一个 agent 成员即 master，
  其会话名仍可叫 `<群>-<成员>`，并把 `supervisor` 指向它。`swarm new` 默认 `--no-master`。
  好处：模型统一（master 也是成员，进拓扑/广场/看板），prompt 走同一套模板。
- **B · 并存**：保留 `cc-<群>` 作"人类可接管的指挥位"，成员里的 master 只是"被指派的主 worker"。
  语义重叠、易混，不推荐。

本文后续按 **A** 展开。

## 5. 引擎：claude 默认 / codex 可选

CLI 新增 `--kind claude|codex`（默认 claude），存入 members.kind，并据此选启动命令：

```bash
# agent.sh：与 _agent_claude_cmd 平行新增
_agent_codex_cmd() {            # 非交互执行模式，prompt 走 heredoc/stdin（具体 flag 实现期确认）
    local task="$1"; _agent_defaults
    local cmd="cd '${AGENT_WORKDIR}' && ${AGENT_CODEX_BIN:-codex} exec"
    [[ -n "$AGENT_MODEL" ]] && cmd+=" --model ${AGENT_MODEL}"
    cmd+=" <<'TTMUX_TASK_EOF'
${task}
TTMUX_TASK_EOF"
    echo "$cmd"
}
```

> 派发：`type=agent` 时按 `kind` 选 `_agent_claude_cmd` / `_agent_codex_cmd`。
> codex 的精确非交互调用方式（`codex exec` / 输入通道 / 权限）在落地时核对其 CLI；
> 仓库已有 `CodexChat.tsx` 读 codex rollout，可佐证 codex 在用。

## 6. 提示词模板

### 6.1 模板引擎选型（Go）—— 结论：**stdlib `text/template`**

| 方案 | 依赖 | 语法 | 评价 |
|------|------|------|------|
| **`text/template`（标准库）** | **0** | `{{.X}}` `{{if}}` `{{range}}` | ✅ **推荐**：零依赖、变量+条件分支+列表全覆盖、与本项目"极简依赖"一致 |
| `text/template` + `Masterminds/sprig` | +1 | 同上 + 100+ helper | 需要 `default`/`trim`/字符串处理再叠，Helm 同款，轻 |
| `pongo2` / `gonja`（Jinja2 兼容） | +1 | Jinja `{{ x }}` `{% if %}` | 仅当**要 Jinja 语法/模板继承/给非开发者编辑**才值得；提示词用不上 |
| `cbroglie/mustache` | +1 | 逻辑无关 `{{x}}` | 太弱，无条件分支，做不了 master/worker 差异 |
| `osteele/liquid` | +1 | Shopify Liquid | 过重 |

**为什么不用 Jinja**：Jinja 是 Python；Go 里只能用它的移植（pongo2/gonja），等于**为了一种语法引入依赖**。
而提示词模板只需要：变量替换、按 `role` 出不同段落（`{{if}}`）、渲染依赖/同伴列表（`{{range}}`）——
`text/template` 全都有。结论：**`text/template`**；若日后需要大量字符串 helper，再加 `sprig`，不引 Jinja。

### 6.2 渲染位置：**后端 Go 渲染，CLI 透传最终 prompt**

```
Web 加成员 ──► 后端(Go)：text/template 渲染 prompt(按 role/kind/上下文)
            ──► ttmux swarm add <群> <成员> --type agent --kind <k> --role <r> … "<已渲染 prompt>"
            ──► CLI：把 prompt 当作 heredoc body 喂给 claude/codex（CLI 不再二次模板）
```

- **CLI 保持"哑"**：收到的 `task` 即最终 prompt，沿用现有 `… <<'EOF' {task} EOF` 机制，零改造。
- **引擎选型落在 Go**：正好回应"golang 用什么"。模板用 `//go:embed` 打进后端二进制（见 §6.4）。
- **CLI 直接用（无后端）**：power-user 路径，`swarm add` 收到的就是原始任务（不模板）。
  *取舍*：这会造成"Web 有模板、CLI 没有"的分叉。缓解见 §9 开放问题（可选：`ttmux-web render-prompt` 子命令供 bash 调用，统一一套）。

### 6.3 模板变量（渲染上下文）

后端从 swarm.db / `swarm status --json` 取：

| 变量 | 含义 |
|------|------|
| `.Swarm` `.Goal` | 蜂群名 / 目标 |
| `.Member` `.Role` `.Kind` | 成员名 / master\|worker / claude\|codex |
| `.Task` | 用户填的原始任务 |
| `.Deps` `.Peers` | 依赖成员列表 / 同群其他成员 |
| `.SkillsDir` | skill 目录（见 §6.5） |
| `.Workdir` | 工作目录 |
| `.MasterName` | 当前 master 成员名（worker 用来知道向谁汇报） |

### 6.4 模板存放与覆盖

- 真源放仓库：`backend/server/prompts/{master,worker}.md.tmpl`，后端 `//go:embed prompts/*.tmpl`。
- 覆盖：`TTMUX_PROMPT_DIR` 指向外部目录则优先读盘（便于不重编辑模板）。
- 一个文件两个具名模板亦可（`{{define "master"}}…{{end}}`）；按 `.Role` 选。

### 6.5 skills 目录如何注入（**别硬编码绝对路径**）

需求是"告诉 agent `/home/ai/codes/ttmux/skills/` 可用，走 swarm 格式"。但：
- 那是**当前机器的仓库路径**，换机即失效；
- agent(claude) 实际**自动加载** `~/.claude/skills/`（`install.sh` 把 `cc-swarm`/`ttmux` 装那儿）。

设计：模板变量 `.SkillsDir`，**默认 `~/.claude/skills`**（claude 自动加载处），可用 `TTMUX_SKILLS_DIR` 覆盖。
模板里点名 **cc-swarm skill = 蜂群协作格式**，让 agent 去读/遵循。

> skills 落盘已实现并统一：**生产** `install.sh`、**开发** `start.sh --dev` 均经
> [`skills/sync-skills.sh`](../../skills/sync-skills.sh) 把仓库 `skills/` 同步到 `~/.claude/skills`
> （或 `TTMUX_SKILLS_DIR`）——单一真源、避免硬编码绝对仓库路径。安装两个技能：
> `ttmux.md`（编排 CLI）、`cc-swarm.md`（蜂群协作）。`cc-swarm.md` 里有「可用 CLI 列表」
> 引用 **`ttmux`** 与 **`chrome`**（浏览器自动化 CLI）；`chrome` 不单独建技能。

### 6.6 模板内容（草案）

**worker.md.tmpl**
```
你是蜂群「{{.Swarm}}」的成员 {{.Member}}（worker）。
蜂群目标：{{.Goal}}
你的任务：{{.Task}}
{{if .Deps}}你依赖：{{.Deps}}（已就绪才轮到你）。{{end}}

协作方式（**务必遵循 swarm 格式**）：
- 可用 skill 在 {{.SkillsDir}}，重点读 **cc-swarm**（蜂群协作规范）。
- 进度/求助/阻塞发广场：`ttmux swarm say {{.Swarm}} --as {{.Member}} --kind note|ask|block <消息>`
- 认领/流转看板卡：`ttmux swarm task <ls|show|move|done> {{.Swarm}} …`
- 有问题找 master「{{.MasterName}}」。
- **干完务必**：`ttmux swarm done {{.Swarm}} {{.Member}}`（解锁下游）。
工作目录：{{.Workdir}}
```

**master.md.tmpl**
```
你是蜂群「{{.Swarm}}」的 master（指挥）。
目标：{{.Goal}}
{{if .Peers}}现有成员：{{.Peers}}{{end}}

职责（**遵循 swarm 格式**）：
- 可用 skill 在 {{.SkillsDir}}，**必读 cc-swarm**（拆解→建卡→派活→巡场→验收的完整规范）。
- 把目标拆成看板卡并派活：`ttmux swarm task add … / assign …`
- 盯广场、解阻塞、做决定：`ttmux swarm feed/say …`
- 验收并推进：成员完成后 `ttmux swarm done {{.Swarm}} <成员>`，整群完成 `ttmux swarm done {{.Swarm}}`。
你自己的起步任务：{{.Task}}
工作目录：{{.Workdir}}
```

## 7. 接口变更

### CLI（`cli/ttmux-cli/lib/swarm.sh` / `agent.sh`）
```
swarm add <群> <成员> --type agent [--kind claude|codex] [--role master|worker] \
                      [--dir/--perm/--model] [--depends-on a,b] <prompt-或-任务>
```
- 新 flag `--kind`（默认 claude）、`--role`（缺省时：无 master 则 master，否则 worker）。
- 存 `kind/role` 入 members；`type=agent` 时按 kind 选启动命令。
- `swarm status --json` 输出加 `kind/role`，供前端拓扑/抽屉展示。

### 后端（`backend/api/swarm.go` + 新 `prompt` 渲染包）
- `SwarmAddMember` body 加 `kind`、`role`（可空，后端补默认）。
- 渲染：取 `swarm status --json` 得 goal/peers/master → `text/template` 渲染 → 作为 `<prompt>` 传给 `swarm add`，并带 `--kind/--role`。
- 新增 `backend/server/prompt/`（embed 模板 + Render(ctx)）。

### 前端（`frontend/src/Swarm.tsx` 加成员 Modal）
- 引擎 Segmented：**🤖 Claude（默认）｜ ✦ Codex**（type=agent 时显示）。
- 角色：自动显示「首个成员 → master（带 tag）」，可手动切；master 已存在时默认 worker。
- 其余字段不变；POST 带 `kind/role`。

## 8. 端到端流程（Web 加成员）

```
表单(kind=claude, role=auto) ─► POST /swarms/:n/members {name,type,task,kind,role,deps,dir}
  ─► 后端: 定 role(首个=master) → 取 goal/peers → text/template 渲染 prompt
  ─► ttmux swarm add <n> <name> --type agent --kind claude --role master "<prompt>"
  ─► CLI: 存(kind,role) → 选 claude → claude -p <<EOF {prompt} EOF  ─► 成员上线
```

## 9. 开放问题 / 取舍

1. **master 调和**（§4）：采 A（master=成员）还是 B（并存）？建议 A——需改 `swarm new` 默认不自动拉 cc。
2. **模板分叉**（§6.2）：CLI 直接 add 不模板。要不要做 `ttmux-web render-prompt`（Go 子命令）让 bash 也能渲染、统一一套？（多一次 Go 调用，但消除分叉。）
3. **codex 调用细节**：`codex exec` 的输入/权限/模型 flag 落地期核对。
4. **prompt 是否入库**：默认不存（省空间、可重算）；要可复现/审计再存 `prompt_rendered`。
5. **角色后续可变**：master 退出/重指派是否支持 `swarm role <群> <成员> master`？（v2）
