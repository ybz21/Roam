# 浏览器镜像：视口 / 传输 / 显示 三层设计

浏览器镜像（`backend/browser/screencast.go` + `frontend/src/BrowserView.tsx`）把后端共享
Chrome 的画面实时投到网页 `<img>` 上。历史上「画面抖动 / 忽大忽小 / 掉成手机布局」的问题全部
源于三个本应独立的概念被耦合在一起。本文档固化分层模型与不变量，后续改动必须遵守。

## 分层模型

| 层 | 是什么 | 谁决定 | 允许何时变 |
|---|---|---|---|
| **渲染视口** | 远端页面的 CSS 视口（`Emulation.setDeviceMetricsOverride` 的 width/height/dpr）。决定网页布局（桌面版/手机版、能看到多少内容） | 连接建立时按观看区推导；或用户显式切设备档 | 仅：新连接、显式切设备、观看区**宽**变化。**其余任何时刻不得变** |
| **传输画质** | `Page.startScreencast` 的 JPEG quality + maxWidth/maxHeight（像素采样密度）| 用户画质档 / 自适应阶梯 | 随时可变（不影响布局与尺寸） |
| **显示映射** | 帧在前端 `<img>` 里怎么摆（object-fit + 显示盒尺寸） | 观看区宽 + 视口纵横比 | 跟观看区**宽**走 |

**用户视角的不变量：清晰度只影响"糊不糊"，永远不影响"内容多大、能看到多少"。**

## 不变量（改代码前先对照）

1. **渲染视口会话内恒定**。桌面视口 = `宽 max(观看区宽, 1280) × 高按观看区宽高比`；手机档用
   机型表。观看区**高度**变化（移动端工具栏显隐、软键盘）**不**触发视口重设——只有宽变才重设。
2. **切画质 / 切设备 / 观看区变化都不重连 WS**。重连 → 新 CDP 会话重设 device metrics → 首帧是
   「视口未稳」的畸形帧（实测 aspect 从 1.54 变 0.97）→ 画面跳。一切运行时切换走 WS 消息：
   `{type:'quality',auto|q}`、`{type:'emulate',...}`。
3. **emulate 消息去重 + 防抖**（前端 `sendEmulate`）：载荷与上次相同不发；stage 抖动 300ms 合并。
4. **显示盒高 = 舞台宽 × (视口高/视口宽)**，与舞台高无关。高度富余→上下留边；不足→裁切。
   都不改变缩放比 → 工具栏显隐时内容不忽大忽小。
5. **强制补帧 = 重发 `Page.startScreencast`**（无需 stop，Chrome 立即吐一帧当前画面）。
   同管线出图 → 与流帧天然像素级一致。**绝不能用 `Page.captureScreenshot`**：它 fromSurface
   出图，模拟视口高于 Chrome 窗口时（手机竖屏 vp 2226 > 窗口 800）会临时改表面尺寸 →
   真实视口被扰动（实测 metadata 跳 1380×2400、蹦出 2760×4800 巨帧）→ 页面重排、画面乱跳。
6. **帧头 w/h 用帧的真实 CSS 视口**（流帧 metadata / 强制帧用记录的 frameW/H），前端坐标换算
   (`mapClientXY`)靠它，不能拿显示尺寸凑。

## 视口自愈（对付外部会话践踏）

CDP 的 `setDeviceMetricsOverride` 作用于**渲染器**、最后写入者赢，且写入会话退出后覆盖仍卡在
渲染器上（见 `cdp-device-metrics-session-scoped` 教训）。共享 Chrome 上还有别的消费者：

- `chrome-cli screenshot --viewport WxH` 会 `setViewportSize()` 改共享 tab 的设备指标，**截完不恢复**；
- agent / DevTools / 其它镜像连接同理。

任何外部写入都会把镜像的视口踩掉 → 布局重排 → 用户看到「点了一下内容突然变大/变小」。

**自愈机制（后端）**：每个流帧带 metadata（真实 CSS 视口）。镜像会话记录自己期望的视口
(expW/expH)；若连续流帧的 metadata 与期望不符（容差 ±2px），且距上次重设 >600ms，就自动重发
本会话的 `setDeviceMetricsOverride` 抢回来。600ms 防抖避免与另一个**活跃**会话打乒乓。

已知限制：两个镜像观看端同时开同一个 tab 且观看区尺寸不同时会互相抢视口（各自自愈）。属于
罕见场景，表现为画面周期性重排；解决办法是关掉一个观看端。

## 历史坑清单（都是真实修过的）

| 现象 | 根因 | 修复 |
|---|---|---|
| 打字时画面忽清忽糊 | 强制帧走全表面分辨率、流帧走降采样，高频交替 | 强制帧 clip+scale 对齐流帧（后被「重发 screencast」替代） |
| 页面加载头两秒画面回跳 | captureScreenshot 比推流慢，旧内容盖新帧 | 流帧活跃时丢弃强制帧（后被「重发 screencast」替代） |
| 手机竖屏点击后画面缩成一块还偏移 | captureScreenshot fromSurface 在「视口 > Chrome 窗口」时扰动真实视口 | 强制补帧改为重发 startScreencast，彻底弃用 captureScreenshot |
| 切画质画面跳、页面忽大忽小 | 切画质触发 WS 重连 → 重设视口 → 畸形首帧 | 画质走 WS 消息不重连 |
| iPad 窄窗掉成手机布局 | 视口宽 = 观看区宽（太窄） | 视口宽下限 1280 |
| 移动端滚动时内容忽大忽小 | 工具栏显隐改观看区高 → 每次都重设视口 + 显示盒跟舞台高走 | 视口只跟宽走；显示盒高按视口比算；emulate 去重防抖 |
| 点击后内容突然变大看不全 | chrome-cli 等外部会话踩掉视口覆盖（1920×980） | 后端视口自愈（本文档） |

## 验证方法

静态页改视口后不重绘、不产新帧，验证必须用**持续重绘的动态页**。前端左上角调试 HUD
（stage / vp / inner / dpr / frame）可实时观察各层数值：`frame` 变而 `vp` 不变 = 有人踩视口；
`stage` 高抖而画面稳 = 显示解耦生效。
